# ADR-003: Схема таблиц в БД RPA

**Дата:** 16.06.2026  
**Статус:** Принято  
**Автор:** — (на основе ТЗ v1.0)  

## Контекст

Роботу нужны две таблицы в БД RPA Data для хранения транзакций и обеспечения
идемпотентности (защита от повторной обработки при retry/сбоях).

## Таблица `breaks` (WF-1)

Хранит статус обработки каждого агента. Уникальный ключ `(login, shift_date)`
позволяет WF-1 при старте пропускать уже обработанных ночников (сверка — шаг 3 ТЗ).

**Ключевые поля:**
- `run_id` — связывает все строки одного запуска для дневной сводки.
- `status` — жизненный цикл: `calculated` → `loaded` (или `skipped` / `error`).
- `be_code` — фиксирует причину пропуска (BE2 — нераспознанный контейнер, BE3 — нет слота).

## Таблица `transactions_main` (WF-2)

Хранит каждый факт переноса активности. Уникальный ключ
`(login, shift_date, activity_name, original_start)` защищает от задвоения
при повторном прогоне (чекпоинт на уровне группы — шаг 34 ТЗ).

**Ключевые поля:**
- `group_name` — группа суммирования FTE (из файла «Прогнозы.xlsx»).
- `original_start / new_start` — исходное и новое время для аудита.
- `status` — `transferred` / `skipped` (нет валидного слота) / `no_candidates` / `no_deficit`.

## Решение по таймзонам

Все `TIMESTAMPTZ`-поля хранятся в UTC (PostgreSQL). Для чтения/отображения
конвертировать в `Europe/Moscow` (`AT TIME ZONE 'Europe/Moscow'`).

## Последствия

- Финальная схема подлежит согласованию с TDR (https://cf.avito.ru/spaces/RPA/pages/905658227).
- Индексы добавить после нагрузочного тестирования если потребуется.
