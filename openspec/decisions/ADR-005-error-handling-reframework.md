# ADR-005: Обработка ошибок по Reframework (отдельный Error Handler)

**Дата:** 16.06.2026  
**Статус:** Принято (конвенция команды RPA)  
**Источник:** Вводная встреча 16.06.2026 (Anastasiya Tretyakova)  

## Контекст

Роботы команды RPA строятся по подходу Reframework. Обработка исключений вынесена
в отдельный воркфлоу Error Handler, а не размазана по основному потоку.

## Решение

### Отдельный воркфлоу `BreaksRecovery_ErrorHandler`

- Имеет **два входа**: ручной проброс (вызов из основного потока) и авто-триггер на ошибке.
- На вход принимает: `error_type`, `error_message`, `workflow_id`, `workflow_name`.
- Логика: нормализация → повторное чтение конфига (проблема могла быть выше) →
  простановка статуса `ERROR` всем необработанным транзакциям с описанием ошибки →
  отправка нотификации в Mattermost.

### Формирование ошибки в основном потоке (Set Edit Fields)

На false-ветке узла заполняется объект:

| Поле | Значение |
|---|---|
| `success` | `false` |
| `error_type` | `Business Error` (BEn) или `System Error` (SEn) |
| `error_message` | Детальное описание + артефакты (процесс, комментарии) |

### Возврат из под-воркфлоу

Каждый под-воркфлоу (Process) возвращает в основной поток:
`success: true/false` + `error_type` + `error_message`. Основной поток присваивает
транзакции финальный статус (`SUCCESS` / `KEPT` / `ERROR` / неопределённый).

### Нюанс нод n8n

Postgres/HTTP-ноды иногда падают, отдавая кривой результат. На этапе разработки —
тестировать поведение, при необходимости включать `Continue On Fail` и вручную проверять
наличие результата перед тем, как вызывать бизнес-ошибку.

### Откат (System Error)

При системной ошибке на середине обработки продумать процесс отката, если следующая
итерация не будет переобрабатывать запись. Если откат не удался — отправить
дополнительное критическое уведомление помимо SE.

## Маппинг на ТЗ

- BE1–BE4 → `error_type = Business Error`, канал `RPA_BN_BreaksRecovery`.
- SE1–SE3 → `error_type = System Error`, канал `RPA_Robot_Prod_Notifications`, email-fallback.

## Последствия

- Создать отдельный воркфлоу `BreaksRecovery_ErrorHandler`.
- В `Main` и `Balance` добавить Set-узлы формирования ошибки на false-ветках и вызовы Error Handler.
- Узел `BE2/BE3 Check` доработать: сейчас просто пишет транзакцию, нужен проброс ошибки + нотификация.
